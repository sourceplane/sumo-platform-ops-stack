# Verification model

The repository uses a split verification model so the stack can scale as a catalog.

## Verify workflow

`verify.yml` splits verification into two lanes:

- a composition matrix that runs `./scripts/verify-composition.sh <name>` for every exported type
- a full catalog lane that runs `./scripts/verify.sh`

The full catalog lane:

- checks generated composition docs are up to date
- checks the generated registry index is current
- checks every composition has example and test fixtures beyond placeholder READMEs
- runs the same OCI dry-run publish path used by release automation

## Docs workflow

`docs.yml` enforces catalog scaffolding generated by `./scripts/generate-readme.sh`.

## Scorecard workflow

`scorecard.yml` rebuilds `registry/index.json` and fails if the checked-in catalog scorecard is stale.

## Release workflow

`release.yml` validates the tag against `stack.yaml`, reruns the full catalog verification, publishes the OCI package, and attaches the registry index to the GitHub release.

## Score model

The generated scorecard currently evaluates:

| Signal | Points |
| --- | ---: |
| Has composition README | 10 |
| Has example assets | 15 |
| Has test assets | 25 |
| Verified by repo automation | 20 |
| Semantic versioned stack | 10 |
| Has usable docs | 10 |
| Has scorecard/security workflow | 10 |

Grades:

- `A+` trusted
- `A` strong
- `B` stable
- `C` experimental
- `D` legacy
